# VDA5050 模拟器：Order 边执行模式设计

**文档状态**：设计稿（实现前评审）  
**适用范围**：`vda5050-simulator` — `SimulationEngine` 对 MQTT `order` 中 `edges` 的解析与运动规划。

---

## 1. 背景与目标

### 1.1 背景

- VDA5050 在每条 **edge** 上提供 **`released`** 字段：`true` 表示允许车辆执行该边；`false` 表示该边尚属 horizon（视野外），车辆不应驶过，需在上一节点停车，等待主站（AOS）通过新的 **order / orderUpdate** 释放后续边。
- 联调与演示中，常见需求是：**不解析 `released`**，按订单给出的边序列**一次性跑完全路径**，便于快速验证路径几何与 state。
- 与 AOS 生产语义对齐时，需要：**严格按 `released` 的连续前缀**执行，在 horizon 边界停车、上报 **state**（`lastNodeId` 为当前停留节点），由 AOS 下发后续指令后再继续。

### 1.2 目标

1. 提供 **两种** 可配置执行模式，通过配置切换；**不包含**「仅收集全部 `released=true` 的边再跨段拼接」的第三种模式（见第 3 节）。
2. **模式一（保留现有联调行为）**：**不关注** `released` 字段，按 `edges` 顺序使用**全部**边建路径并跑完。
3. **模式二（新增）**：**STRICT_PREFIX_RELEASED** — 仅执行从第一条边起的 **连续 `released=true` 前缀**；遇第一条 `released=false` 的边则**不驶入**，车辆停在**该边起点节点**（即上一 released 边的终点），`driving=false`，并上报 state；待 AOS 再次下发释放更多边后继续。

---

## 2. 术语

| 术语 | 含义 |
|------|------|
| **边序列** | `order.edges` 按 `sequenceId` 排序后的有序列表。 |
| **连续 released 前缀** | 从边序列第一条边开始，连续满足 `released == true` 的最长前缀；若第一条边即为 `released=false`，则前缀为空。 |
| **Horizon 停车点** | 严格模式下，执行完「连续 released 前缀」后车辆所在节点；即第一条未释放边的 `startNodeId`（与上一段终点一致）。 |

---

## 3. 不包含的模式（刻意不实现）

以下模式**不**作为产品选项保留：

- **AGGREGATE_RELEASED**：从全订单中**收集所有** `released=true` 的边（非连续），再按 `sequenceId` 排序后拼接路径。  
  - 会在中间存在 `released=false` 的边时**跳过**该段路径，产生与地图拓扑不一致的「跳跃」执行，**不符合**常见 VDA5050 horizon 语义，也不符合「全路径联调」诉求。  
  - **结论**：不实现、不文档化为可选枚举值。

---

## 4. 模式定义

### 4.1 `IGNORE_RELEASED`（不关注 released）

**语义**

- 解析 `order.edges` 时**忽略**每条边上的 `released` 字段（是否存在、true/false 均不参与决策）。
- 按 `sequenceId` 排序后，对**全部**边建立运动段并依次执行，直至最后一条边的终点。

**适用场景**

- 快速联调、几何验证、不关心主站分步释放策略时。

**State 行为**

- 仍忽略 `released` 建路径；**多段路径**（如 1→2→3）时，**仅在到达最终终点**时因**路径节点变化**发 MQTT `state`；**途经中间节点**（如到达 2）不因该变化发 `state`，但内部会更新去重快照。**不抑制** AOS **及时报文**的应答：例如 `instantActions` 仍触发带 `actionStates` 的 `state`（策略中优先于中间点规则）；首帧、续单抑制、立即发布等既有分支亦优先。

**实现要点（相对当前代码）**

- 若当前实现通过 `collectReleasedEdges` 仅保留 `released=true` 的边，则在本模式下应改为：**使用排序后的全边列表** 构建 `RouteFollower`，不再按 `released` 过滤。
- **行驶中途再次收到 `order`**（例如及时报文后 AOS 下发仍含完整前缀的新单）时，需 **裁掉已驶过的前导边** 并 **对齐段内进度**，避免从第一条边重建导致「回头」。详见专项设计：[IGNORE_RELEASED 中途换单前缀裁剪设计](./IGNORE_RELEASED_中途换单前缀裁剪设计.md)。

---

### 4.2 `STRICT_PREFIX_RELEASED`

**语义**

1. 将 `edges` 按 `sequenceId` 排序。
2. 计算 **连续 released 前缀**（见第 2 节）。  
   - **`released` 字段缺失**时的语义由配置项 `releasedFieldMissingAsReleased` 固定（见第 5 节），实现时**不要**再混用其它隐式约定。
3. 仅对前缀内的边建路径并执行。  
4. **前缀为空**（例如第一条边即 `released=false`）：不沿本订单移动；`driving=false`；记录日志。**位姿**应与「当前仿真器已确立的车辆状态」一致（连接后初始位姿、或上一订单结束位姿），避免与 `state` 中 `agvPosition` 矛盾；是否额外强制推送 state 可与 AOS 约定，必要时在实现阶段定义「进入等待 horizon」时的发布策略。

**停车与 AOS 协同**

- 车辆到达 **horizon 停车点** 后：`driving=false`；**state** 中 **`lastNodeId`** 为当前停留节点（OpenTCS 侧命名经现有 `mapNameCodec` 转换，与现有一致）。
- AOS 根据 state 下发新的 **order** 或 **orderUpdate**（更高 `orderUpdateId`、释放后续边）；模拟器在收到新订单后，从当前 `lastNodeId` 与新区块对齐，继续执行下一段**连续 released 前缀**。**orderUpdate** 是否局部替换边图、与旧 `orderId` 的衔接方式，须与 `CommAdapter` 实际报文一致后再写入实现说明或 README。

**适用场景**

- 与 YFAOS 生产逻辑一致，验证「分步释放 + 到站再下发」流程。

## 5. 配置项（草案）

在 `application.yaml` 的 `simulation` 段增加，例如：

```yaml
simulation:
  # IGNORE_RELEASED — 忽略 released，按 edges 全量顺序跑完
  # STRICT_PREFIX_RELEASED — 仅执行连续 released 前缀，在 horizon 边界停车等下发
  orderExecutionMode: IGNORE_RELEASED

  # 可选：仅在 STRICT_PREFIX_RELEASED 下生效
  # true: edge 缺少 released 字段时视为 true
  # false: 缺少 released 字段时视为 false
  releasedFieldMissingAsReleased: true
```

**默认值建议**

- `orderExecutionMode: IGNORE_RELEASED` — 与「保留当前直接跑完」的联调习惯一致。  
- `releasedFieldMissingAsReleased: true` — 若选 STRICT 且与旧 JSON 兼容为主（可与后端字段约定一并确认）。

## 6. 行为对照表

| 项目 | `IGNORE_RELEASED` | `STRICT_PREFIX_RELEASED` |
|------|-------------------|-------------------------|
| 是否读取 `released` | 否 | 是 |
| 边列表来源 | 排序后的**全部**边 | 排序后的**连续 released 前缀** |
| 中间存在 `false` 边 | 仍沿全序列行驶 | 在第一条 `false` 边之前停车 |
| 与 AOS 分步下发 | 不模拟 | 模拟 |

## 7. 非功能与边界

- **cancelOrder**：两种模式下均保持现有取消语义（清空 route、订单相关字段等）。
- **orderId / orderUpdateId 更新**：STRICT 模式下续跑策略与 AOS 下发的节点/边是否全量或增量相关，**实现阶段**与 `CommAdapter` 报文对齐后补充至本文或 README。
- **README**：实现完成后在 `vda5050-simulator/README.md` 增加简短指向本设计文档的链接与配置示例（路径：`docs/订单边执行模式设计.md`）。

## 8. 设计评审：可改进与后续细化

以下项在实施前可再收敛，避免实现与联调预期漂移：

| 方向 | 说明 |
|------|------|
| **空前缀时的 state** | 除 `driving=false` 外，是否必须立即发一帧 state（即使 `lastNodeId` 未变），建议在联调 checklist 与 AOS 确认。 |
| **orderUpdate 语义** | VDA5050 允许订单更新；需明确模拟器是否只支持「全量 order 替换」、是否支持局部边列表更新，与 `aos-backend` 行为一致。 |
| **观测性** | STRICT 下在 horizon 边界停车时，建议打结构化日志（`orderId`、停驻节点、`released` 断点 `sequenceId`），便于与 MQTT 抓包对照。 |
| **测试矩阵** | 至少覆盖：全 `true`、首条 `false`、中间单条 `false`、缺失 `released` 字段 × 两种 `releasedFieldMissingAsReleased`；与 `IGNORE_RELEASED` 对照同一 JSON。 |
| **规范对齐** | 若对外交付，可补充 VDA5050 文档中与 `released`/horizon 对应的章节引用（版本与节号随所用规范固定）。 |
